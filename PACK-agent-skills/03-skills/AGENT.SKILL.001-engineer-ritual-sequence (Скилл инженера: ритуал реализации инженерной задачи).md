# AGENT.SKILL.001 — Скилл инженера

## Назначение

Канонический ритуал выполнения инженерной задачи в экзокортексе без дрейфа.

## Последовательность

1. Сначала согласование:
   - какой именно РП открываем;
   - какой агент исполняет задачу;
   - нужно ли дописывать skill текущему постоянному агенту до старта работ.
2. Диагностика по фактам: логи, health-check, конкретные файлы/entrypoint.
3. Сверка с Цереном: как это реализовано в рабочем upstream-эталоне.
4. Сверка с целевым эталоном нашего контура: как должно быть у нас.
5. Оценка риска перед правками: что может сломаться и как проверим после.
6. Фиксация РП и контекста:
   - `inbox/WP-*.md`,
   - `docs/WP-REGISTRY.md`,
   - `PACK-exocortex-engineering/04-work-products/ENG.WP.*.md`.
7. Реализация правок только после шагов 1-6.
8. Пост-проверка: health-check и целевые проверки сценария.
9. Закрытие: truthful итог, коммит, push, запись в `inbox/INBOX-TASKS.md`.

## Iteration Closure Gate

Инженер не имеет права переходить к следующему шагу, следующему инкременту или следующему РП,
пока текущая итерация не закрыта полным циклом:

1. `manual check`
2. `git status`
3. `commit`
4. `push`
5. повторный `git status`

Если после этого в дереве остаётся хвост от текущей итерации, итерация считается **не закрытой**.

### Truthful language rule

Инженеру запрещено использовать формулировки:

- `сделано`
- `закрыто`
- `готово`

если:

- нет `push`,
- нет повторной проверки `git status`,
- или в дереве остался его собственный незавершённый хвост.

### Generated artifacts rule

Generated-артефакты (`__pycache__`, пересобранные временные файлы, bot-output, generated reports)
не могут оставаться «на потом».

Они должны:

- либо входить в commit осознанно как часть результата,
- либо очищаться сразу,
- но не переноситься в следующую итерацию по инерции.

## Acceptance Gate

Если инженерный slice затрагивает внешний или интеграционный контур, где возможна реальная проверка,
инженер не имеет права закрывать итерацию как `done` без живой верификации ключевого цикла.

Под живой верификацией понимать:

- не только создание артефакта;
- но и проверку ключевых операций жизненного цикла;
- и подтверждение, что контур реально работает, а не только "выглядит настроенным".

### Примеры

- `Google Drive`: создать -> изменить -> переместить -> удалить.
- `Google Calendar`: создать -> прочитать -> изменить -> удалить.
- `Gmail`: если slice про почту, минимум нужен `send` и проверка факта отправки.
- runtime агент: запуск entrypoint -> проверка статуса -> проверка лога -> подтверждение результата сценария.

### Truthful status rule

Если не было нормального проверочного прогона, truthful статус может быть:

- `partial`
- `verification pending`
- `architecture only`

но не `done`.

## Жёсткие запреты

- Нельзя открывать новый РП без явного согласования с владельцем.
- Нельзя нанимать sidecar / explorer / временного агента без явного согласования с владельцем.
- Нельзя подменять постоянного доменного агента временным агентом только ради скорости.
- Если у постоянного агента не хватает capability, сначала дописывается его skill, и только потом он запускается на задачу.
- Если согласование пропущено, инженер обязан:
  - остановиться,
  - честно зафиксировать ошибку,
  - вернуть процесс к шагу согласования, а не продолжать инерцией.
- Нельзя переходить к следующему инкременту с собственным грязным деревом.

## Правило для предметных агентов

- Один предметный контур -> один основной агент.
- Инженер не должен “таскать новых агентов” вместо усиления основного доменного агента.
- Для повторяющихся задач правильный путь:
  - уточнить дефект,
  - дописать skill основному агенту,
  - прогнать именно этого агента,
  - зафиксировать, чему он научился в этой итерации.

## Критерий завершения

- Критичная проблема устранена или эскалирована с фактическим блокером.
- Результат подтверждён проверками.
- Хронология сохранена в реестре и инженерных артефактах.
- Ритуал согласования не был нарушен, либо нарушение явно зафиксировано и исправлено до продолжения.
- Текущая итерация закрыта циклом `manual check -> commit -> push -> clean tree`.
