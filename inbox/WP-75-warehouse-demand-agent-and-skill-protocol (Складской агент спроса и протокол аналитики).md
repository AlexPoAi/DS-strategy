# Контекст

Складской контур уже умеет делать техническую обработку, но нужно закрепить предметный слой:
- отдельный агент со специализацией по складу;
- чёткий протокол анализа входящих документов Жанны;
- карточки с управленческими рекомендациями (закупка, распродажа, риск дефицита).

# Цель

Сделать повторяемый цикл `документ -> аналитика -> карточка -> Telegram-выжимка`, где каждая поставка данных от Жанны превращается в actionable решение.

# Scope

1. Добавить профильного агента `Warehouse Demand Analyst` в реестр агентства.
2. Зафиксировать skill-протокол: остатки, продажи, ABC, маржа, рекомендации.
3. Подготовить и сохранить свежий аналитический артефакт по складу на текущих данных.
4. Сверить 24/7 контур на VPS и зафиксировать фактический runtime-статус.

# Acceptance

1. В агентстве есть карточка нового складского агента + запись в реестре.
2. В `PACK-warehouse` создан новый управленческий артефакт по свежим данным.
3. В карточке есть 4 блока решений: `срочно заказать`, `планово`, `сократить/распродать`, `проверить вручную`.
4. В `WP-63`/хронологии зафиксирован фактический VPS-статус и найденные блокеры.

# Ритуальное уточнение исполнителя (2026-04-20)

Следующий bounded slice по складу должен выполняться уже не в смешанном режиме
`инженер + доменная интерпретация по ходу`, а с явным разделением ролей:

- `Environment / Code Engineer`
  - чинит intake, pipeline, Telegram delivery, Drive routing, registry и архитектуру;
- `Warehouse Demand Analyst`
  - читает накладные, остатки, продажи, ABC;
  - собирает решение по закупке;
  - группирует по поставщикам;
  - даёт руководителю короткий управленческий отчёт без шумовых артефактов;
  - фиксирует каких данных не хватает.

### Исполнитель следующего складского среза

Primary executor: `Warehouse Demand Analyst`

Support executor: `Environment / Code Engineer` только если доменный агент упирается
в технический blocker пайплайна.

### Truthful anti-drift note

Если складской отчёт снова превращается в перечисление файлов, карточек, csv и
технических путей, это значит, что работу опять выполняет инженерный слой вместо
доменного агента склада.

## Verdict от нанятого Warehouse Demand Analyst (2026-04-20)

### Что сейчас не так

- Текущий `WH.REPORT.002` остаётся технической sync-сводкой, а не отчётом для руководителя.
- В нём видны pipeline-счётчики и артефакты, но нет финальных блоков решения:
  - `срочно заказать`
  - `планово заказать`
  - `не заказывать / сократить`
  - `проверить вручную`
- `WH.WP.007` уже даёт supplier map, но decision-layer ещё не дотянут:
  нет полной связки `low-stock -> supplier -> quantity -> deadline -> risk`.

### Каким должен быть финальный формат

Короткое manager-ready сообщение в Telegram должно быть не про документы, а про действия:

1. `Срочно заказать`: SKU / у кого / сколько / до какой даты / риск
2. `Планово заказать`: SKU / поставщик / горизонт
3. `Не заказывать / сократить`: что в избытке
4. `Проверить вручную`: аномалии, спорные SKU, missing contacts

Отдельно сверху:
- одна executive summary строка:
  `сколько критичных SKU`, `сколько поставщиков задействовано`, `где главный риск`.

### Следующие 5 шагов

1. Связать дефицитные SKU из остатков с supplier map из `WH.WP.007`.
2. Для каждого критичного SKU определить `у кого заказать`, `контакт`, `lead time`, `order_deadline`.
3. Добавить расчёт `сколько заказать` через `par level / reorder point`, а не просто через перечисление дефицита.
4. Разобрать `pending_owner_review` сессии в decision-layer, чтобы вместо очереди были 4 готовых блока решений.
5. Только после этого собирать финальную Telegram-выжимку для руководителя.
