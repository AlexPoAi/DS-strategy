# WP-64: Рефактор инженерного контура и план закрытия хвостов

## Роль и найм агента
- Нанят профильный агент: **Environment Engineer** (рефактор runtime/governance слоя).
- Сопряжённый исполнитель: **Code Engineer** (реализация технических правок по активным хвостам).

## Контекст
- Инженерный контур накопил много параллельных задач и частично устаревших статусов.
- Часть задач уже фактически закрыта, но трекеры не везде синхронизированы.
- Из-за этого повышается риск повторного ремонта и расфокуса.

## Цель
Собрать единый и короткий closeout-план по инженерному контуру с приоритетами, дедупликацией задач и измеримым acceptance по каждому активному WP.

## План работ
1. **Нормализация статусов**
   - Пройти `INBOX-TASKS`, `ENGINEERING-HIT-LIST`, `ENGINEERING-CHRONOLOGY`.
   - Пометить устаревшие/закрытые инженерные пункты как `done`.
   - Убрать дубли и ложные `priority-tomorrow`.

2. **Приоритизация хвостов**
   - Зафиксировать `critical` контур: `WP-61`, `WP-62`, `WP-63`, `CI Node24/validate`.
   - Для каждого пункта задать ясный next-slice и критерий завершения.

3. **Closeout-матрица**
   - Материализовать короткую матрицу: `задача -> блокер -> действие -> артефакт -> критерий done`.
   - Привязать матрицу к weekly execution-порядку.

4. **Синхронизация хронологии**
   - Записать результат в `ENGINEERING-CHRONOLOGY`.
   - Убедиться, что новый агент может войти в контур за 5-10 минут без потери контекста.

## Acceptance
- В `INBOX-TASKS` нет явных инженерных дублей или устаревших приоритетов.
- Есть один согласованный список `critical/high` действий на текущую неделю.
- У каждого активного WP есть next-slice и измеримый критерий завершения.
- Хронология обновлена и отражает фактическое состояние контура.
