# РП#22: Стабилизация рабочей среды экзокортекса

## Почему это стало приоритетом №1

Пользователь явно остановил обычную работу по предметным задачам до тех пор, пока среда экзокортекса не станет устойчивой. Наблюдаемый паттерн: процессы запускаются, но не доводятся до конца; статус-артефакты иногда расходятся с реальным runtime; часть сценариев является интерактивной, но запускается как будто автоматическая; контур закрытия и контекст сессии ведут себя нестабильно.

## Ключевое различение

Проблема **не только в токене**.

Токен и auth-helper были лишь одним из симптомов. Реальная проблема — в стыках между несколькими слоями:
1. auth/helper/proxy слой
2. выбор модели и совместимость с runtime
3. wrapper-скрипты агентов
4. prompt-сценарии (автоматические vs интерактивные)
5. status-artifacts / health-check / daily-report
6. close-task.sh / SESSION-CONTEXT / protocol-close

## Почему у Церена работает стабильнее

Шаблон Церена проще и ближе к upstream. В локальном экзокортексе поверх шаблона добавлены кастомные слои:
- helper-based auth через `~/.config/aist/env`
- custom proxy / base URL
- отдельные wrapper-скрипты
- session-open gate
- собственный close-task контур
- status-artifacts и пользовательские русские экраны состояния

Из-за этого падают не отдельные инструменты, а интеграционные швы между ними.

## Цель

Собрать **минимальный устойчивый рабочий контур**, в котором:
- Claude CLI стабильно запускается
- strategist / extractor / scheduler дают правдивый статус
- автоматические сценарии доходят до конца
- интерактивные сценарии не маскируются под автоматические
- close-task и SESSION-CONTEXT корректно фиксируют завершение
- пользователь может доверять зелёному статусу среды

## Артефакт

1. Карта всех сценариев экзокортекса: что автоматическое, что интерактивное
2. Минимальный зелёный runtime-контур
3. Исправленный контур статусов и закрытия
4. Ясный операторский протокол: как запускать день и как проверять, что система реально здорова

## Минимальный зелёный контур

### Автоматические сценарии
- strategist: `morning`
- strategist: `note-review`
- extractor: `inbox-check`
- synchronizer: `daily-report`
- synchronizer: `health-check`

### Интерактивные сценарии
- strategist: `session-prep`
- strategist: `strategy-session`
- любые сценарии, где требуется явное подтверждение пользователя внутри процесса

## Фазы работы

### Фаза 1. Инвентаризация сбоев
- Зафиксировать все точки, где runtime и статус расходятся
- Составить список сценариев, которые не должны запускаться как fire-and-forget
- Проверить, какие ошибки уже исправлены, а какие только маскируются статусом

### Фаза 2. Разделение автоматических и интерактивных процессов
- Явно развести сценарии стратега на 2 класса: automation-safe и human-in-the-loop
- Запретить считать интерактивные сценарии частью фонового автоматического контура

### Фаза 3. Truthful status layer
- Проверить, что AGENTS-STATUS / SchedulerReport / SESSION-OPEN показывают реальность, а не устаревший снимок
- Убрать ложную зелень и ложные ошибки

### Фаза 4. Контур закрытия
- Исправить автозапись в SESSION-CONTEXT
- Исправить вывод close-task.sh
- Разобраться с `protocol-close.md` и привести runtime к документации

### Фаза 5. Приёмочный тест end-to-end
- Открытие дня
- extractor inbox-check
- strategist note-review
- ручной запуск session-prep
- ручной запуск strategy-session
- close-task
- повторный health-check

## Критерии готовности

- [ ] Claude CLI стабильно работает через helper-only схему
- [ ] Все автоматические сценарии доходят до конца без ручного вмешательства
- [ ] Интерактивные сценарии явно обозначены как интерактивные
- [ ] AGENTS-STATUS и SchedulerReport отражают фактическое состояние
- [ ] close-task и SESSION-CONTEXT работают без склеек и дублирования
- [ ] Пользователь может безопасно приостановить предметную работу до зелёного контура и затем возобновить её
