# WP-95 Phase 2 Readiness Gate v1

## Назначение

Этот документ нужен, чтобы переход к `Phase 2` не происходил по импульсу, усталости или желанию “раз уж нашли человека, давайте навесим всё остальное”.

Его задача:

- удержать границу между `Phase 1` и `Phase 2`;
- не открыть расширение роли раньше подтверждённой передачи ядра;
- дать единый gate для решения `идём / не идём`.

## Truthful framing

`Phase 2` в `WP-95` означает не “любое следующее движение”, а расширение роли за пределы warehouse/accounting ядра.

Потенциальный контур `Phase 2`:

- Telegram-коммуникация;
- часть `Google Drive` / документов;
- кухня;
- найм;
- дополнительные admin-функции.

Этот контур нельзя открывать, пока `Phase 1` не стал по-настоящему рабочим.

## Главный принцип gate

`Phase 2` нельзя открывать, если `Phase 1` ещё держится на:

- скрытом ownership Жанны;
- нестабильном handoff;
- ручном дожиме нового человека;
- красивых словах без устойчивого cycle.

## Обязательные условия входа в gate

Перед обсуждением `Phase 2` должны быть собраны:

1. `Phase 1 Launch Verdict`
2. factual summary по первому cycle
3. понимание, какие блоки ядра уже реально держатся
4. понимание, какие блоки всё ещё risk / fail

Без этого дальше идти нельзя.

## Условия, при которых `Phase 2` можно рассматривать

Все условия ниже должны быть одновременно выполнены:

1. Verdict по запуску `Phase 1` — `success`
2. Новый человек понимает bounded scope и удерживает его без критичного дрейфа
3. Жанна не остаётся скрытым ежедневным owner ядра
4. Нет критичного провала по:
   - остаткам;
   - инвентаризации;
   - накладным;
   - `Saby`;
   - `Честному знаку`;
   - недостачам
5. Руководитель не находится в режиме постоянного ручного дожима
6. Есть ясность, какой именно блок `Phase 2` открываем первым

## Условия, при которых `Phase 2` открывать нельзя

Достаточно любого одного пункта:

1. Verdict по `Phase 1` = `partial` или `fail`
2. Жанна всё ещё фактически держит ядро
3. Новый человек не держит рутину без постоянных напоминаний
4. Роль уже начала размываться
5. Хочется “просто добавить ещё функций”, потому что есть свободное место
6. Неясно, какой именно кусок `Phase 2` открывается и зачем

## Красные флаги перед расширением роли

- `Phase 1` визуально спокойна только потому, что Жанна страхует контур
- новый человек делает ядро с ошибками, но обсуждение уже перескочило к Telegram/кухне/найму
- `Google Drive` хотят передать целиком, хотя даже базовый upload routine ещё не стабилен
- `Phase 2` используется как способ “добить объём работы”, а не как управляемое расширение роли
- нет отдельного bounded slice для первого блока `Phase 2`

## Какой первый `Phase 2` slice выглядит разумно

После `Phase 1` логично открывать не весь хвост разом, а только один bounded block.

Наиболее вероятные кандидаты:

1. `Google Drive` и операционные документы
2. Telegram-коммуникация с командой
3. отдельный admin-layer кухни

Не рекомендовано:

- открывать всё вместе;
- смешивать `Telegram + кухня + найм + документы` в один переход;
- считать это “естественным продолжением” без нового WP-slice.

## Решение gate

### `ready`

Ставим, если:

- `Phase 1` реально передана;
- есть устойчивый cycle;
- можно открыть один следующий bounded block без разрушения ядра.

### `not ready`

Ставим, если:

- `Phase 1` ещё требует стабилизации;
- ownership ещё не перешёл полностью;
- любое расширение увеличит риск collapse.

На текущем состоянии `2026-04-20`:

- `Phase 2` обсуждать методологически можно;
- `Phase 2` открывать operationally ещё нельзя;
- сначала нужен реальный verdict по запуску `Phase 1`.
