# WP-101 — Убрать дубли daily-telegram и закрепить один источник истины статуса

## Почему открыто

Утром 2026-04-22 в Telegram пришли два ежедневных отчёта экзокортекса:

1. `09:03` — локальный `launchctl + codex` отчёт с корректным зелёно-жёлтым статусом.
2. `08:15` — VPS `systemd + claude` отчёт с красным статусом и ошибками `strategist morning / note-review`.

Это создаёт ложную картину состояния, потому что два разных runtime-контура
отправляют `daily-telegram-report` в один и тот же чат.

## Truthful factual finding

- Локальный контур (`Mac / launchctl`) отправил отчёт в `09:03:09`.
- VPS-контур (`systemd`) отправил другой отчёт в `08:15:19`.
- На VPS подтверждены ошибки:
  - `Not logged in · Please run /login`
  - `Canonical protocol route incomplete ... under /root/Github/memory`
- Значит, это не один дрейфующий отчёт, а **две независимые отправки из двух разных сред**.

## Skeleton задачи

### Шаг 1 — Диагностика по фактам

Подтвердить:
- какие именно runtime-контуры имеют право слать `daily-telegram-report`;
- какие chat/token/env используются локально и на VPS;
- есть ли общий бизнес-контракт, что daily-report должен идти только из одного места.

### Шаг 2 — Сверка с эталоном

Проверить:
- как у Церена организован единственный источник ежедневного статуса;
- какой target-state правильный для нас:
  - `one source of truth`,
  - либо `prod-chat + tech-chat` разделение,
  - но не два конкурирующих отчёта в одном месте.

### Шаг 3 — Оценка риска до правок

Нельзя случайно поломать:
- product/cloud runtime на VPS;
- ежедневный локальный статус-отчёт;
- transport-layer `notify.sh`;
- другие Telegram сценарии, которые легитимно живут на VPS.

### Шаг 4 — Принятие решения

Нужно выбрать один из вариантов:

1. `Local authority`
- ежедневный отчёт экзокортекса шлётся только с локального `launchctl/codex`;
- VPS больше не шлёт `daily-telegram-report` в этот чат.

2. `VPS authority`
- ежедневный отчёт шлётся только с VPS;
- локальный daily-telegram выключается.

3. `Split channels`
- локальный идёт в основной chat;
- VPS идёт только в техchat/monitoring channel.

### Шаг 5 — Фундаментальный фикс

После выбора архитектуры:
- отключить лишний sender;
- закрепить source-of-truth в runtime/policy/документации;
- сделать anti-regression guardrail, чтобы dual-send не вернулся.

## Предварительный инженерный verdict

На текущий момент safest-default выглядит так:
- **основной daily status = локальный `launchctl/codex`**
- **VPS daily status = отключить или увести в отдельный техканал**

Но это ещё не финальное решение до сверки с эталоном и оценки риска.

## Критерий завершения

РП не считается завершённым, пока:
- в основной Telegram-чат не остаётся ровно **один** ежедневный отчёт;
- не зафиксировано, какой runtime является каноническим sender;
- не подтверждено, что отключение дубля не ломает другие cloud/product сценарии.
