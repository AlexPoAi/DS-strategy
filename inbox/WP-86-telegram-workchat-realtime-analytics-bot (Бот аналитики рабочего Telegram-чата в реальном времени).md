# WP-86 — Бот аналитики рабочего Telegram-чата в реальном времени

## Контекст
Нужен отдельный бот, который читает сообщения в рабочем Telegram-чате кофейни, агрегирует сигналы по темам (склад, продажи, персонал, инциденты, поставки), и выдаёт руководителю управленческую аналитику и actionable-выжимку.

## Feasibility verdict
**Реализуемо.**

Ограничения Telegram, которые нужно принять сразу:
1. Bot API не отдаёт полную старую историю чата автоматически.
2. Бот читает новые сообщения только после добавления в чат и корректной настройки.
3. Для чтения групповых сообщений нужен бот в чате + выключенный privacy mode (BotFather) + согласованная политика доступа.

## Цель продукта
- Получать в near real-time сигналы из рабочего чата.
- Давать руководителю короткие отчёты:
  - что произошло;
  - где риски;
  - какие решения принять сегодня;
  - какие данные/действия запросить у ответственных.

## Scope (Wave 1)
1. Подключить бота к рабочему чату (read-only ingest).
2. Поток событий: `Telegram updates -> ingest -> queue -> analysis -> manager digest`.
3. Классификация сообщений по доменам:
   - Склад/закупки
   - Продажи/гости
   - Персонал/графики/замены
   - Инциденты/эскалации
4. Два типа отчёта:
   - Мгновенный alert (критичные темы)
   - Сводка за период (утро/день/вечер)

## Архитектура (target)
1. **Collector:** Telegram bot/webhook listener.
2. **Storage:** журнал событий (SQLite/Postgres + минимальная PII-маска).
3. **Analyzer:** извлечение тем, задач, рисков, владельцев.
4. **Publisher:** Telegram-дайджест руководителю + ссылки на артефакты в PACK.

## Риски и policy
1. Команда должна быть уведомлена, что чат анализируется ботом.
2. Нельзя хранить лишние персональные данные.
3. Нужна role-based выдача отчётов (руководитель/менеджер/операционный контур).

## Definition of Done (Wave 1)
1. Бот стабильно принимает новые сообщения из рабочего чата.
2. Формирует минимум 1 ежедневную сводку с actionable пунктами.
3. Формирует критичные alerts по ключевым триггерам.
4. Сводка содержит владельца действия и срок.

## План работ
1. Согласовать доступ и policy (кто видит что, что храним).
2. Развернуть ingest-сервис на VPS 24/7.
3. Подключить классификацию и шаблон manager-report.
4. Провести пилот 7 дней.
5. Довести до production-контура.
