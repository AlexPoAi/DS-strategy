# Контекст

Нужно смотреть склад не как “папку с отчётами”, а как отдельную доменную область бизнеса VK Coffee:
- с собственным контрактом данных,
- с регулярной аналитикой,
- с автоматизированными решениями по закупке/перезатарке/марже.

# Цель

Собрать актуальный baseline склада (`что уже есть`) и зафиксировать roadmap автоматизации (`что делать дальше`) с приоритетами и измеримым эффектом.

# Что уже есть (зафиксировано)

1. Базовый pipeline `sync -> cards -> bot -> telegram` (WP-62).
2. Registry + DLQ + idempotency + governance-контур (WP-63).
3. Синк папки Жанны и обработка не только Google Sheets, но и `.xlsx/.csv` (WP-65).
4. Реестр документов домена и набор report-cards по остаткам/продажам/накладным.
5. VPS-таймер 24/7 для складского цикла (`vk-warehouse-full-loop.timer`).

# Ключевые gap-зоны

1. Неполная materialization ABC (файлы есть, карточки/решения не всегда выходят стабильно).
2. Нет стабильного SLA качества аналитики (из чего формируется “готовый управленческий вывод”).
3. Нет автоматического reorder-модуля (минимальные пороги, прогноз, заказ-лист).
4. Нет регулярного plan-fact по экономическому эффекту (маржа, оборачиваемость, замороженные деньги).
5. Нет единой бизнес-панели склада для владельца (weekly snapshot + рекомендации).

# Scope

1. Зафиксировать baseline артефактов и текущую готовность домена.
2. Описать target-state склада как автономного бизнес-контура.
3. Сформировать roadmap автоматизаций по волнам:
   - Wave 1: быстрые эффекты (1-2 недели),
   - Wave 2: управленческая зрелость (2-6 недель),
   - Wave 3: прогнозирование и инвестиционная связка (6+ недель).

# Acceptance

1. Есть доменный документ с baseline + target + roadmap.
2. У каждой инициативы есть owner, KPI, risk и срок.
3. Минимум 3 инициативы Wave 1 готовы к запуску сразу.
