# WP-72 — Ритуал согласования для каждого нового рабочего продукта

## Проблема

В текущем поведении агента есть опасный дрейф: после корректного открытия общей рабочей сессии агент может начать **новый отдельный рабочий продукт** как будто это просто продолжение уже подтверждённой работы.

Из-за этого нарушается обязательная граница:
- сессия открыта;
- но новый `WP` / новый bounded slice ещё **не согласован отдельно**.

Реальный инцидент 2026-04-19:
- рабочая сессия `DS-strategy` была открыта корректно;
- затем был фактически открыт новый slice `WP-56` с чтением backlog и редактированием артефактов;
- отдельный ритуал `Роль / Работа / РП / Метод / Оценка / Модель` перед этим не был пройден.

## Почему это критично

1. Ломается дисциплина WP Gate и ритуала согласования.
2. Агент может незаметно открывать новые РП внутри уже идущей сессии.
3. Пользователь теряет контроль над точкой, в которой анализ превращается в отдельную работу и начинает менять файлы.
4. Ошибка особенно опасна в длинных стратегических сессиях, где легко спутать «продолжение разговора» и «новый рабочий продукт».

## Цель

Сделать так, чтобы **каждый новый рабочий продукт** внутри активной сессии требовал отдельного явного ритуала согласования до чтения файлов, анализа и редактирования.

## Что нужно сделать

1. Зафиксировать точную семантику:
   - открытие сессии ≠ открытие нового РП;
   - новый `WP`, новый bounded audit, новый fix-slice = новый ритуал.
2. Найти все места в opening/work protocol и repo-инструкциях, где это различение сейчас не закреплено или допускает двусмысленность.
3. Добавить явный guardrail:
   - если агент меняет фокус с одного РП на другой;
   - если агент создаёт новый `WP-*`;
   - если агент начинает bounded slice, не объявленный в предыдущем ритуале;
   - он обязан остановиться и пройти отдельное согласование.
4. Обновить инженерную память и/или протоколы так, чтобы это не зависело от «помнить в голове».
5. Добавить post-check сценарий:
   - привет → открытие сессии;
   - подтверждение;
   - затем попытка открыть новый WP;
   - агент должен сам остановиться и запросить отдельное подтверждение.

## Acceptance

- В протоколах и/или инструкциях явно различаются:
  - открытие сессии;
  - открытие нового рабочего продукта.
- Есть правило: новый `WP` / новый bounded slice не может стартовать по инерции от уже открытой сессии.
- Есть проверяемый сценарий, в котором агент корректно останавливается перед новым РП и заново проходит ритуал.
- Инцидент 2026-04-19 зафиксирован как root example, а не теряется в переписке.

## Риски

- Если зафиксировать правило слишком общо, агент снова будет трактовать новый slice как «просто продолжение».
- Если правило внедрить только в один файл, возможен повторный drift между `CLAUDE.md`, memory-протоколами и repo-специфичными инструкциями.

## Следующий срез

1. Сверить текущий opening/work contract с этим инцидентом.
2. Определить минимальный набор файлов, где правило должно быть закреплено.
3. Внести изменение и отдельно прогнать post-check сценарий на живой сессии.
