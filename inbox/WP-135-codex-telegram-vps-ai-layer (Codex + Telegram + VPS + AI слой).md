# Codex + Telegram + VPS + AI слой

## Контекст

Пользователь хочет завтра, `2026-05-06`, отдельно обсудить, как сейчас устроена связка `Codex / Telegram / VPS / AI`.

Текущая боль:

- Telegram-бот находится на VPS и должен работать `24/7`;
- непонятно, работает ли сейчас `Codex` с Telegram напрямую или только через отдельный бот/рантайм;
- нет нормальной возможности оплатить OpenAI API: карта не принимается;
- нужно понять, можно ли сделать рабочий ИИ-слой через подписку, альтернативный API, локальную модель, прокси-провайдера или другую архитектуру.

## Главный вопрос

Как сделать так, чтобы VPS реально работал с Telegram-ботом и ИИ-слоем стабильно, без ручной зависимости от ноутбука и без блокировки из-за проблемы оплаты OpenAI API?

## Что нужно разобрать

### 1. Фактическое состояние

- Какой Telegram-бот сейчас запущен на VPS.
- Через какой `systemd` / `pm2` / `launchd` / docker-контур он живёт.
- Есть ли у него сейчас доступ к RAG, базе знаний, embeddings и ответам модели.
- Какие переменные окружения и токены реально стоят на VPS.
- Что сейчас ломается: Telegram webhook/polling, RAG, OpenAI API, embeddings, deploy, auth или бизнес-логика.

### 2. Codex и Telegram

- Может ли `Codex` сейчас быть оператором Telegram-бота напрямую.
- Если прямой маршрут невозможен или небезопасен, какой правильный мост: Telegram bot -> backend -> task/inbox -> Codex session / agent runtime.
- Какие действия можно доверить боту автоматически, а какие должны уходить в inbox на подтверждение.

### 3. AI слой без нормальной оплаты OpenAI API

Проверить варианты:

- OpenAI API с другим платёжным способом / другим аккаунтом / бизнес-картой.
- Официальная подписка ChatGPT: проверить, можно ли её использовать для API-задач или это отдельный контур.
- Альтернативный OpenAI-compatible провайдер как временный шлюз.
- Локальная модель на VPS или на домашнем Mac как fallback.
- Гибрид: дешёвая модель для простых ответов, сильная модель только для сложных задач.

Важно: не строить решение на серой автоматизации веб-интерфейса, если она нестабильна или нарушает условия сервиса.

### 4. Архитектура результата

Нужна схема:

- Telegram как пользовательский интерфейс.
- VPS как 24/7 runtime.
- AI provider adapter как заменяемый слой.
- RAG / база знаний как отдельный слой.
- Очередь задач / inbox как safety gate для серьёзных действий.
- Observability: healthcheck, логи, ежедневный статус, алерт при падении.

## Нужные входные данные

- SSH-доступ / путь к текущему VPS-проекту.
- Название и токен текущего Telegram-бота, без раскрытия секрета в документе.
- Текущие `.env` переменные на VPS.
- Логи последнего запуска.
- Что пользователь хочет делать через Telegram: вопросы, отчёты, задачи, управление агентами, документы, продажи, склад.
- Ограничение по бюджету в месяц на AI.

## Формат результата

1. Карта текущего состояния: что уже работает, что сломано, чего нет.
2. Ответ: может ли сейчас `Codex` работать с Telegram и через какой маршрут.
3. 2-3 архитектурных варианта: safe / pragmatic / ambitious.
4. Отдельный verdict по оплате API и подписке: что реально, что нет, что проверить.
5. План внедрения на 1-2 вечера без большого переписывания.
