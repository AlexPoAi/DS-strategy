# WP-104 — Жёсткий gate закрытия итерации инженера

## Почему открыто

Во время сегодняшней работы по `WP-97` инженерный цикл несколько раз разорвался:

- локальные правки и generated-артефакты остались в дереве;
- часть итераций не была закрыта цепочкой `manual check -> commit -> push -> clean tree`;
- в конце дня в рабочем дереве накопился большой хвост изменений, который не был сразу признан как незакрытая итерация.

Это не разовая забывчивость, а дефект инженерного протокола:
в skill не было жёсткого exit gate, запрещающего переход к следующему инкременту без чистого дерева.

## Skeleton задачи

### Шаг 1 — Truthful diagnosis

Подтвердить дефект по фактам:
- незавершённая локальная правка может остаться в дереве;
- generated-файлы копятся между итерациями;
- инженер может сообщить об окончании шага раньше, чем выполнены `commit/push/clean-tree`.

### Шаг 2 — Сверка с эталоном

Нормальный инженерный цикл должен закрываться так:

1. manual check
2. `git status`
3. `commit`
4. `push`
5. повторный `git status`
6. только после этого следующий инкремент

### Шаг 3 — Risk assessment

Нельзя:
- требовать идеальной чистоты за счёт удаления чужих изменений;
- считать незакоммиченные generated-файлы «мелочью»;
- писать `сделано / закрыто / готово`, если остался собственный хвост.

### Шаг 4 — Фундаментальный фикс

Нужно материализовать rule-layer:
- в skill инженера;
- в инженерном work-product;
- в реестре/контексте;
- с отдельным названием `Iteration Closure Gate`.

## Что сделано

- В `AGENT.SKILL.001` добавлен отдельный раздел `Iteration Closure Gate`.
- Добавлен прямой запрет на переход к следующему шагу без:
  - `manual check`
  - `git status`
  - `commit`
  - `push`
  - повторного `git status`
- Добавлено truthful правило языка:
  - нельзя писать `сделано / закрыто / готово`, если нет `push` и чистого дерева по своему слою.
- Добавлено правило про generated-артефакты:
  - либо входят в commit осознанно,
  - либо удаляются/очищаются сразу,
  - но не остаются «на потом».

## Критерий завершения

РП считается закрытым, когда:

- правило materialized в skill инженера;
- правило записано в инженерный repair-layer;
- рабочее дерево после фикса не содержит моего незавершённого хвоста;
- следующий день можно тестировать уже новый протокол, а не устную договорённость.
