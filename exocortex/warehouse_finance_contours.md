# Два контура VK Coffee: Кладовщик и Экономист

## Контур кладовщика (PACK-warehouse)

**Локация:** `/Users/alexander/Github/VK-offee/PACK-warehouse/`

**Источник данных:** Google Drive папка **"Отчёты для бота/Новые документы"** 
- ID папки: `1Jg1Zgj2_ueTV6-NQamAU8XCPalFNhO8W`
- Жанна загружает: остатки, продажи, выручка, накладные (PDF), ABC анализ, каталог

**Запуск:** Только вручную
```bash
bash PACK-warehouse/tools/warehouse_full_loop.sh
```

**Что делает:**
1. **Intake** — скачивает файлы из Drive в `knowledge-base/Отчёты для бота/Новые документы/`
2. **Классификация** — определяет тип: остатки, продажи, выручка, накладные, ABC
3. **Обработка** — анализирует дефициты, топ продажи, маржу, ABC-приоритет
4. **Нормализация** — создает WH.CARD.* карточки в `02-domain-entities/report-cards/`
5. **Отчёт** — формирует WH.REPORT.* и WH.SESSION.001 в `04-work-products/`
6. **Доставка** — отправляет Telegram digest со ссылками
7. **Архив** — перемещает файлы Drive: "Новые документы" → "Обработано" (ID: `1WanukzWeuqJgUQ7N8YkG9oC23-DIRGjT`)

**Выходные артефакты:**
- `WH.CARD.*` — карточки отчётов (остатки, продажи, ABC по каждому документу)
- `WH.REPORT.002` — статус синхронизации (всегда обновляется)
- `WH.SESSION.001` — очередь решений (срочно заказать, плановая закупка, не заказывать, проверить)
- Telegram digest с ключевыми рекомендациями

**Ответственный:** Warehouse Demand Analyst агент
**Протокол:** `WH.METHOD.001-zhanna-biweekly-intake-and-processing-skill`
**Архитектура:** `WH.WP.006-warehouse-full-cycle-architecture-2026-04-19`

## Контур экономиста (DS-finance-private)

**Локация:** `/Users/alexander/Github/DS-finance-private/business-finance/`

**Источник данных:** Google Drive папка **"Отчёты для бота"** (ТЕ ЖЕ файлы что и кладовщик)
- Подпапки: Продажи, Выручка, Накладные, Каталог
- Специальная ID: Продажи (`1g2Pq-cYVT2cSkd9DwBCV-EJBHYDvzHw9`)

**Запуск:** Через финансового консультанта (координатор DS-finance)
```
financial-consultant → vk_finance_analyst (INTAKE-слой)
```

**Что делает:**
1. **Intake** — скачивает Продажи, Выручка, Накладные, Каталог из Drive в `data-layer/`
2. **Трансформация** — считает метрики: выручка по точкам, маржа %, динамика
3. **Нормализация** — нормализует точки (turgeneva / lugovaya / samokisha)
4. **Анализ** — выявляет аномалии (просадки >20%), считает среднюю/день
5. **Finance View** — создает `finance-view-YYYY-MM-DD.md` в `views/` с P&L, маржой, трендом
6. **Передача** — handoff Finance View в `financial-consultant` для verdict (freeze/keep/invest)

**Выходные артефакты:**
- `finance-view-YYYY-MM-DD.md` — полный Finance View с выручкой, маржой, аномалиями по точкам
- **Не публикуется в git** (финансовые данные, только локально)
- Потом передаётся в decision-layer для verdict

**Ответственный:** VK Finance Analyst агент
**Координатор:** Financial Consultant (orchestrator DS-finance)
**Слои:** data-layer → transformation-layer → decision-layer

## Взаимодействие двух контуров

### Проблема: Конфликт на файлах

```
Жанна кидает в Drive "Новые документы"
         ↓
Кладовщик скачивает, обрабатывает, ДВИЖЕТ в "Обработано"
         ↓
Экономист тоже хочет файлы — но они уже в "Обработано"!
```

### Решение: Единый оркестратор

**НИКОГДА не трогать файлы на Drive ДО того как обработали их оба!**

```
Жанна кидает в Drive "Новые документы"
         ↓
[ЭТАП 0: INTAKE] Скачать файлы в локальное хранилище
   sync-google-sheets.py → ~/VK-offee/knowledge-base/Отчёты для бота/Новые документы/
   ❌ Файлы на Drive остаются в "Новые документы" (не двигаем пока!)
         ↓
[ЭТАП 1: КЛАДОВЩИК] Читает из локального хранилища
   bash PACK-warehouse/tools/warehouse_full_loop.sh
   → WH.CARD, WH.SESSION → PACK-warehouse/
         ↓
[ЭТАП 2: ЭКОНОМИСТ] Читает из локального хранилища (ТЕ ЖЕ файлы!)
   vk_finance_analyst → Finance View → DS-finance-private/business-finance/views/
         ↓
[ЭТАП 3: АРХИВ] Только ПОСЛЕ обоих
   Переместить на Drive: "Новые документы" → "Обработано"
   Обновить реестры в обоих контурах
```

## Разделение данных (кто читает что)

| Тип файла | Кладовщик | Экономист | Источник |
|-----------|-----------|-----------|---------|
| **Остатки** | ✅ читает | ❌ | Сверка баланса |
| **Продажи** | ✅ читает (для ABC) | ✅ читает | Выручка по точкам |
| **Выручка** | ✅ читает | ✅ читает | Кассовые отчёты |
| **Накладные (PDF)** | ✅ читает (цены, поставщики) | ✅ читает (COGS) | Приход + себестоимость |
| **ABC анализ** | ✅ читает | ✅ читает | Приоритет + product mix |
| **Каталог** | ✅ читает (цены) | ✅ читает (маржа) | SKU, маржа % |

**Важно:** Оба читают ТЕ ЖЕ файлы, но извлекают разные срезы для своих доменов.

## Нормализация (единый источник истины)

**Точки (обязательная одинаковая нормализация):**
```yaml
точка_slugs:
  тургенева: turgeneva
  карьерный: lugovaya
  самокиша: samokisha
```

Это должно быть в `drive-folders-vkoffee.yaml` — ОДИН файл, который используют оба агента.

**Типы маржи (разные!)**:
- Кладовщик: `catalog_margin` (из Каталога, расчётная)
- Экономист: `actual_margin` (из Накладных, фактическая COGS)

Они НЕ должны совпадать, это нормально. Просто нужно ясно помечать какая маржа где.
